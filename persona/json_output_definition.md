# JSON output definition

This document explains the JSON summary files generated by the simulator data pipeline.

## Covered files

- `data/processed/simulated_events_validation_test.json`
- `data/processed/simulated_events_validation.json`
- `data/processed/event_split_summary_test.json`
- `data/processed/event_split_summary.json`

The test and production versions use the same schema.

## 1. simulated_events_validation schema

This file is a validation summary for the raw simulated event log.

### Top-level keys

- `mode`
  - runtime mode used to generate the summary
  - example: `test`, `production`
- `input_file`
  - source event CSV path used for validation
- `row_count`
  - total number of event rows in `simulated_events`
- `unique_users`
  - number of unique users appearing in the event log
- `unique_sessions`
  - number of unique sessions appearing in the event log
- `event_type_counts`
  - row counts grouped by event type
  - expected keys: `search`, `view`, `cart`, `purchase`
- `active_persona_counts`
  - row counts grouped by the session-level active persona used at generation time
- `missing_search_query_rows`
  - number of `search` rows where `query_text` is empty
- `missing_item_rows`
  - number of `view`, `cart`, or `purchase` rows where `article_id` is empty
- `first_timestamp`
  - earliest timestamp found in the log
- `last_timestamp`
  - latest timestamp found in the log
- `elapsed_seconds`
  - execution time used to build the validation summary

### Usage

- quick sanity check for event scale
- event-type balance check
- missing field check before downstream training

## 2. event_split_summary schema

This file is a summary of the time-based train/valid/test split.

### Top-level keys

- `mode`
  - runtime mode used to generate the split
- `input_file`
  - source simulated event file used for splitting
- `ratios`
  - configured split ratios
  - expected keys: `train`, `valid`, `test`
- `train`
  - summary block for the training split
- `valid`
  - summary block for the validation split
- `test`
  - summary block for the test split
- `elapsed_seconds`
  - execution time used to build the split outputs

### Split block keys

The `train`, `valid`, and `test` blocks share the same schema.

- `split`
  - split name
- `row_count`
  - total number of rows in the split
- `unique_users`
  - number of distinct users in the split
- `unique_sessions`
  - number of distinct sessions in the split
- `event_type_counts`
  - row counts grouped by event type inside that split
- `first_timestamp`
  - earliest timestamp included in that split
- `last_timestamp`
  - latest timestamp included in that split

### Usage

- check whether the split followed the expected 8:1:1 rule
- compare event balance across train/valid/test
- verify that time ordering is preserved

## Notes

- these JSON files are metadata summaries, not the raw event logs themselves
- the raw event content remains in `simulated_events*.csv`
- the train/valid/test row-level data remains in `train_events*.csv`, `valid_events*.csv`, and `test_events*.csv`
